# Тонкие случаи проекта (WPF + EF Core) — примеры с подвохами

Справочник конкретных ситуаций, которые разбирались в работе над проектом. Каждый блок: **в чём подвох** + **правильный код**. Использовать как образцы, чтобы не наступать на те же грабли.

---

## 1. Конструктор `testContext()`

**Подвох:** EF Core Power Tools генерирует контекст только с конструктором `DbContextOptions` (под DI). `new testContext()` не скомпилируется. И нельзя «вернуть» контекст из конструктора (`return ctx;` — ошибка, конструктор ничего не возвращает).

**Правильно** — параметрless-конструктор + `OnConfiguring` в отдельном partial-файле (scaffold его не затрёт):
```csharp
// testContext.Partial.cs
using Microsoft.EntityFrameworkCore;
namespace test2.Models;

public partial class testContext
{
    public testContext() { }

    protected override void OnConfiguring(DbContextOptionsBuilder b)
    {
        if (!b.IsConfigured)   // не мешать, если контекст создан через DI с options
            b.UseSqlServer("Server=.;Database=shoe_store;Trusted_Connection=True;TrustServerCertificate=True");
    }
}
```
Альтернатива — цепочка к базовому конструктору: `public testContext() : base(new DbContextOptionsBuilder<testContext>().UseSqlServer("...").Options) { }`.

## 2. ComboBox по внешнему ключу внутри карточки (`DataTemplate`)

**Подвох:** внутри `DataTemplate` `DataContext` — это сам объект (Product), а список вариантов (`Categories`) лежит на контроле. Нужно «дотянуться» вверх через `RelativeSource`. Плюс: `Categories` должно быть **публичным свойством**, и на контроле обязателен `DataContext = this`, иначе combobox пустой.

```xml
<ComboBox DisplayMemberPath="Name"
          ItemsSource="{Binding DataContext.Categories,
                        RelativeSource={RelativeSource AncestorType=ListBox}}"
          SelectedValue="{Binding IdCategory}"
          SelectedValuePath="Id"/>
```
```csharp
public ObservableCollection<Category> Categories { get; set; }   // свойство, не поле!
// ...
Categories = _context.Categories.Local.ToObservableCollection();
DataContext = this;   // без этого ItemsSource не найдёт Categories
```
- `MappingName`/`SelectedValue` = поле-ключ `IdCategory` (туда пишется id);
- `DisplayMemberPath="Name"` = что видно; `SelectedValuePath="Id"` = что сохраняется.
- Если карточки в `ItemsControl`, а не `ListBox` → `AncestorType=ItemsControl`.

## 3. Сортировка

**Подвох:** `Products.Sort()` или `Employee.Sort()` — неправильно: `Employee` это тип, а `ObservableCollection.Sort()` не существует. Сортировать надо **представление**, не саму коллекцию.

```csharp
var view = CollectionViewSource.GetDefaultView(MainList.ItemsSource);
view.SortDescriptions.Clear();          // убрать прошлые правила, иначе наложатся
view.SortDescriptions.Add(new SortDescription("Quantity", ListSortDirection.Ascending));
// view.Refresh();  // не нужен — при изменении SortDescriptions пересортируется сам
```
- `"Quantity"` — имя свойства **строкой** (компилятор опечатку не поймает).
- По убыванию — `ListSortDirection.Descending`.
- Сортировка живёт на том же view, что и фильтр — **совмещается** с ним и сохраняется при поиске.

## 4. Поиск и фильтр (вместе, в реальном времени)

**Подвох:** `Products = Products.Where(...)` — двойная ошибка: (1) `Where` возвращает `IEnumerable`, в `ObservableCollection` не присвоить; (2) это **разрушает** коллекцию (товары не вернутся) и **не обновляет UI** (`ItemsSource` держит старую ссылку). Правильно — фильтровать **представление**, коллекцию не трогать.

**Поиск + фильтр в одном предикате** (требование «применять совместно»):
```csharp
private string _searchText = "";
private int? _filterSupplierId = null;   // null = «Все поставщики»

private void ApplyFilter()
{
    var view = CollectionViewSource.GetDefaultView(MainList.ItemsSource);
    view.Filter = obj =>
    {
        if (obj is not Product p) return false;

        bool matchesSearch = string.IsNullOrEmpty(_searchText)
            || (p.Name?.Contains(_searchText, StringComparison.OrdinalIgnoreCase) ?? false)
            || (p.Description?.Contains(_searchText, StringComparison.OrdinalIgnoreCase) ?? false)
            || (p.IdCategoryNavigation?.Name?.Contains(_searchText, StringComparison.OrdinalIgnoreCase) ?? false);

        bool matchesSupplier = _filterSupplierId == null || p.IdSupplier == _filterSupplierId;

        return matchesSearch && matchesSupplier;
    };
}

private void Search(object sender, TextChangedEventArgs e)
{
    _searchText = ((TextBox)sender).Text;
    ApplyFilter();
}
private void SupplierFilter_Changed(object sender, SelectionChangedEventArgs e)
{
    _filterSupplierId = (SupplierBox.SelectedItem as Supplier)?.Id;   // «Все» = null
    ApplyFilter();
}
```
Тонкости:
- по полям через FK искать по **`Navigation.Name`** (текст), а не по числовому id;
- `?.` и `?? false` — чтобы не падать на `null` (например, пустое `Description`);
- `StringComparison.OrdinalIgnoreCase` — без учёта регистра;
- очистка: пустой текст / `_filterSupplierId == null` → условие пропускает всё.

## 5. Загрузка списка с несколькими навигациями

**Подвох:** одна навигация — один `Include`. Для нескольких — цепочка. Для «навигация внутри навигации» — `ThenInclude`. Для списка лениво **не** грузить (N+1 запросов, тормоза).

```csharp
_context.Products
    .Include(p => p.IdCategoryNavigation)
    .Include(p => p.IdManufacturerNavigation)
    .Include(p => p.IdSupplierNavigation)
    .Include(p => p.IdUnitNavigation)
    .Load();
Products = _context.Products.Local.ToObservableCollection();
MainList.ItemsSource = Products;

// вложенная навигация:
_context.Orders.Include(o => o.OrderItems).ThenInclude(oi => oi.IdProductNavigation).Load();
```

## 6. ObservableCollection vs List

**Подвох:** «не удаётся преобразовать ObservableCollection в List» — значит поле объявлено `List<T>`, а присваивается `ToObservableCollection()`. Для привязки и живого add/remove нужен `ObservableCollection`.
```csharp
using System.Collections.ObjectModel;
public ObservableCollection<Product> Products { get; set; }   // не List
```
`List` сохраняется в БД нормально, но при добавлении/удалении UI сам не обновится.

## 7. Показ связанного значения вместо id

**Подвох:** `{Binding IdCategory}` покажет число, `{Binding IdCategoryNavigation}` — тип объекта. Нужно поле навигации:
```xml
<TextBlock Text="{Binding IdCategoryNavigation.Name}"/>
```
Работает только если навигация загружена через `Include`, иначе пусто.

## 8. INotifyPropertyChanged / `[ObservableProperty]`

**Подвох:** `[ObservableProperty]` вешается на **поле**, не на свойство; DataAnnotations пробрасываются через `[property: ...]`; для подписи поля брать `[Display(Name=...)]`, а не `[DisplayName]` (грид/форма читают именно `Display`).
```csharp
[ObservableProperty]
[property: StringLength(100)]
[property: Display(Name = "Имя")]
private string _name = null!;
```
Ручной вариант (атрибуты остаются на свойстве):
```csharp
private string _name = null!;
[StringLength(100)]
public string Name { get => _name; set => SetProperty(ref _name, value); }
```
Важно: **редактирование в форме и сохранение работают и без INPC** (биндинг пишет в объект, EF трекает сам). INPC нужен только для живого обновления UI из кода / в нескольких местах сразу. То же для коллекций: сохранение связей идёт без `ObservableCollection`, она нужна только для живого обновления списка при add/remove.

## 9. Навигация между экранами

**Подвох:** `Session.Window.MainContent = new SomeControl()` не работает — надо менять `.Content`, а не сам `ContentControl`. И `Window.GetWindow(this)` в **конструкторе** контрола вернёт `null` (ещё не в дереве).
```csharp
// MainWindow:
public MainWindow()
{
    InitializeComponent();
    Session.Window = this;            // окно доступно всем экранам
    Navigate(new ResgisrationControl());
}
public void Navigate(UserControl v)
{
    if (MainContent.Content is UserControl c) _history.Push(c);
    MainContent.Content = v;          // .Content!
}

// из любого экрана:
Session.Window.Navigate(new tableControl());
```

## 10. UserControl, а не Page

**Подвох:** если экран создан как `Page`, при вставке в `ContentControl` падает `ParentOfPageMustBeWindowOrFrame`. Все экраны для подмены — `UserControl`. Ещё: у `UserControl` нет свойства `Title` (уберёшь — уйдёт ошибка). Базовый класс в XAML (`<UserControl>`) и в code-behind (`: UserControl`) должны совпадать, иначе «разделяемые объявления указывают разные базовые классы».

## 11. Вход (логин)

**Подвох:** переключать экран нужно **только при успехе**, а не до проверки; гость = `CurrentUser = null` (а не «вход под клиентом»); `PasswordBox` читается через `.Password`, не `.Text`.
```csharp
private void Login(object sender, RoutedEventArgs e)
{
    Session.CurrentUser = _context.Users
        .Include(u => u.IdRoleNavigation)
        .FirstOrDefault(u => u.Login == LoginText.Text && u.Password == PasswordText.Password);

    if (Session.CurrentUser == null)
    {
        MessageBox.Show("Неверный логин или пароль.", "Ошибка входа",
            MessageBoxButton.OK, MessageBoxImage.Warning);
        return;   // остаёмся на входе
    }
    Session.Window.Navigate(new tableControl());
}
```

## 12. Форма добавления/редактирования (одно окно на оба режима)

**Подвох:** не делать две формы. Одно окно: `product == null` → добавление. Модальный `ShowDialog()` сам блокирует открытие второго окна редактирования. `DialogResult = true` → список обновляется.
```csharp
public ProductEditWindow(testContext ctx, Product product = null)
{
    InitializeComponent();
    _context = ctx;
    _isNew = product == null;
    _product = product ?? new Product();
    IdBox.Visibility = _isNew ? Visibility.Collapsed : Visibility.Visible;  // ID скрыт при добавлении
    DataContext = _product;
}
private void Save_Click(object s, RoutedEventArgs e)
{
    if (_product.Price < 0 || _product.Quantity < 0) { /* MessageBox */ return; }
    if (_isNew) _context.Products.Add(_product);   // ID присвоит БД
    _context.SaveChanges();
    DialogResult = true; Close();
}
// открытие: if (new ProductEditWindow(_context, p).ShowDialog() == true) ReloadProducts();
```

## 13. Удаление со связью

**Подвох:** нельзя удалить товар, который есть в заказе — проверить **до** удаления.
```csharp
if (_context.OrderItems.Any(oi => oi.IdProduct == product.Id))
{
    MessageBox.Show("Нельзя удалить товар, который присутствует в заказе.",
        "Удаление невозможно", MessageBoxButton.OK, MessageBoxImage.Warning);
    return;
}
_context.Products.Remove(product);
_context.SaveChanges();
ReloadProducts();
```

## 14. Подсветка карточек по условию (IValueConverter)

**Подвох:** условную раскраску удобнее всего делать конвертером, привязывая фон **к самому объекту** (пустой Binding):
```csharp
public class StockBackgroundConverter : IValueConverter
{
    public object Convert(object value, Type t, object p, CultureInfo c)
    {
        if (value is Product pr)
        {
            if (pr.Quantity == 0) return Brushes.LightBlue;                       // нет на складе
            if (pr.Discount > 15) return (Brush)new BrushConverter().ConvertFrom("#2E8B57"); // скидка >15%
        }
        return Brushes.Transparent;
    }
    public object ConvertBack(object v, Type t, object p, CultureInfo c) => throw new NotImplementedException();
}
```
```xml
<Window.Resources><local:StockBackgroundConverter x:Key="StockBg"/></Window.Resources>
...
<Border Background="{Binding Converter={StaticResource StockBg}}"> ... </Border>
```

## 15. Раскладка (частые ошибки)

- **`ListBox`/`SfDataGrid` не оборачивать в `ScrollViewer`** — у них свой скролл; внешний даёт бесконечную высоту и контент вылезает/перекрывает. Список — в строку `*`, поиск/кнопки — в строку `Auto`.
- **`Width="*"` у элемента не работает** — звёздочка только у `ColumnDefinition`/`RowDefinition`. «Во всю ширину» = без `Width` + `HorizontalAlignment="Stretch"`. Карточки во всю ширину списка = `ListBox HorizontalContentAlignment="Stretch"`.
- **Фиксированная высота режет содержимое** — `Border Height="35"` обрежет всё ниже 35px; не задавать высоту, пусть растёт по содержимому.
- **Клик по «пустому» месту не ловится** — у контейнера нет фона; добавить `Background="Transparent"` (прозрачный, но кликабельный). Объект клика брать из `(sender as FrameworkElement).DataContext`.
- **`Grid.Row`/`Grid.Column` действуют только на прямых детей `Grid`** — на вложенных элементах игнорируются.
